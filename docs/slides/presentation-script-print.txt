[ Slide 2 — 문제 정의 & 비전 / 약 20초 ]

일단, 덱빌딩 로그라이크의 핵심 재미는 "이번 런엔 어떤 빌드로?" 라고 설명할 수 있겠는데요.
그런데 모바일에서는 이 재미를 가볍게 즐길 수가 없습니다.

왜냐하면 완성도 높은 작품들은 대부분 PC·콘솔 전용이고,
카드와 시스템이 복잡해 진입 장벽이 높으며, 메타 성장이 없으면 반복 플레이 동기도 약합니다.

그래서 저희는 Flutter(플러터) 크로스 플랫폼, 미니멀 규칙,
그리고 런마다 XP(엑스피)를 쌓아 레벨업하면 카드와 유물이 영구 해금되는 구조로
이 세 가지를 해결합니다.

[ Slide 5 — 아키텍처 설계 / 약 25초 ]

아키텍처는 4계층으로 분리했습니다.

Presentation(프레젠테이션)은 화면만,
Application(애플리케이션)의 Riverpod Notifier(리버팟 노티파이어)가 상태를 관리하고,
Domain(도메인)은 순수 Dart(다트)로 게임 로직,
Data(데이터)는 SharedPreferences(쉐어드 프리퍼런스)를 통해 데이터를 저장하고 불러옵니다.

의존 방향은 Presentation이 Application을,
Application이 Domain과 Data를 각각 의존합니다.
Domain에는 Flutter(플러터)와 Riverpod(리버팟) import(임포트)를 금지해
테스트 가능성을 극대화했습니다.

덕분에 화면 없이 ProviderContainer(프로바이더 컨테이너) 하나만으로
전체 게임 로직을 단독 테스트할 수 있습니다.

[ Q1 ] "왜 Flutter(플러터)를 선택했나요?"

두 가지 이유입니다.

첫째, 단일 코드베이스로 Android(안드로이드)와 iOS(아이오에스)를 동시에 지원합니다.
1인 개발에서 두 플랫폼을 따로 개발하면 공수가 두 배가 됩니다.
React Native(리액트 네이티브)도 같은 장점이 있지만,
카드 드래그·애니메이션 같은 커스텀 UI를 구현할 때
React Native(리액트 네이티브)는 JS Bridge(제이에스 브릿지)를 거쳐야 해서 제약이 생깁니다.
Flutter(플러터)는 자체 렌더링 엔진을 써서 네이티브 컴포넌트에 의존하지 않기 때문에
카드 UI를 자유롭게 구현할 수 있습니다.

둘째, Hot Reload(핫 리로드)입니다.
카드 레이아웃을 바꿀 때마다 앱을 재시작하지 않고 바로 확인할 수 있어서
짧은 개발 기간에 이터레이션 속도가 크게 올라갔습니다.

----------------------------------------------------------------


[ Q2 ] "이 화면의 데이터는 어디서 오나요?"

Presentation(프레젠테이션) 계층의 위젯이
Riverpod(리버팟)의 ref.watch(레프 왓치)로 Provider(프로바이더)를 구독합니다.
예를 들어 전투 화면은 battleProvider(배틀 프로바이더)를 구독해서,
BattleNotifier(배틀 노티파이어)가 상태를 바꾸면 위젯이 자동으로 다시 그려지는 구조입니다.
데이터 자체는 Domain(도메인)의 BattleEngine(배틀 엔진)이 계산하고,
Application(애플리케이션)이 이를 받아 상태로 emit(이밋)합니다.

  (더 설명 요청 시:
   세 역할로 나눠서 생각하면 됩니다.
   BattleEngine(배틀 엔진)은 주방입니다. HP가 얼마 깎이는지 실제 숫자를 계산합니다.
   BattleNotifier(배틀 노티파이어)는 서빙 직원입니다. 계산 결과를 화면에 가져다줍니다.
   BattleScreen(배틀 스크린)은 손님입니다. 받은 숫자를 그냥 화면에 표시합니다.
   ref.watch(레프 왓치)는 "숫자 바뀌면 나한테 알려줘"라고 등록해두는 겁니다.)

----------------------------------------------------------------


[ Q3 ] "지금 가장 큰 문제는?"

두 가지입니다.

첫째, 테스트 커버리지입니다.
Domain(도메인) 계층 80% 목표를 아직 달성하지 못했고, 발표 전까지 확보해야 합니다.

둘째, 일정 압박입니다.
UI(유아이)-ViewModel(뷰모델) 통합, 테스트, QA(큐에이), 발표 준비가 모두 남아 있습니다.
다만 이 두 가지는 WBS(더블유비에스) 수립 당시부터 위험 등록부에 식별해둔 항목이고,
우선순위에 따라 순서대로 처리하고 있습니다.

----------------------------------------------------------------


[ Q4 ] "어디서부터 어디까지 AI가 만들었나요?"

AI를 적극적으로 활용했습니다.
문서 작성, 코드 초안 생성, 아키텍처 설계 검토에 사용했습니다.
AI가 생성한 코드는 테스트를 먼저 작성하고, 테스트를 통과할 때만 병합하는
TDD(티디디, Test-Driven Development) 방식으로 검증했습니다.

----------------------------------------------------------------


[ Q5 ] "본인이 직접 짠 부분은 어떤 거예요?"

게임 규칙과 아키텍처 구조를 직접 설계했고,
카드 데미지 계산과 상태이상 적용 로직을 설계하고 검토했습니다.
AI가 코드 초안을 써줬지만,
무엇을 만들지, 어떻게 연결할지는 저희가 판단하고 결정했습니다.

----------------------------------------------------------------


[ Q5 세부 ] "코드 흐름 설명해보세요" / "데미지 계산이 어떻게 돼요?"

카드 한 장을 썼을 때 4단계로 흐릅니다.

  1단계 — 화면 (BattleScreen: 배틀 스크린)
  유저가 카드를 드래그하면 이 한 줄만 실행됩니다.
  "ref.read(battleProvider.notifier).playCard(card)"
  → 읽을 때: 레프 리드, 배틀프로바이더 노티파이어, 플레이카드
  화면은 Application(애플리케이션)에 전달만 합니다.

  2단계 — Application (BattleNotifier: 배틀 노티파이어)
  "_engine.playCard(card)"  → 게임 엔진에 위임 (읽을 때: 엔진 플레이카드)
  "state = _fromEngine()"   → 결과를 화면 상태로 변환해서 emit(이밋)
                               (읽을 때: 스테이트 프롬엔진)

  3단계 — Domain (BattleEngine: 배틀 엔진)
  · 에너지 부족하면 false(폴스) 반환
  · 에너지 차감, 카드를 손패에서 버림덱으로 이동
  · Strike(스트라이크)면 몬스터 데미지, Defend(디펜드)면 플레이어 블록 획득
  · Weak(위크) 상태면 데미지 × 0.75 (소수점 버림)
  · Vulnerable(벌너러블) 상태 몬스터는 받는 피해 × 1.5
  · 몬스터 HP 0이면 승리 처리
  Flutter(플러터) import(임포트) 없음 → 화면 없이 테스트 가능

  4단계 — 화면 재빌드
  새 상태가 emit(이밋)되면 BattleScreen(배틀 스크린)이 자동으로 다시 그려집니다.

  한 줄 요약:
  "화면은 전달만, Application은 중계만, 게임 규칙은 Domain에만 있습니다."

  데미지 숫자 (외울 것):
  · 취약(Vulnerable: 벌너러블) → 받는 데미지 × 1.5
  · 약화(Weak: 위크)           → 주는 데미지 × 0.75 (소수점 버림)
  · 블록                       → 턴 종료 시 0으로 리셋

[ Q6 ] "왜 4계층 아키텍처를 선택했나요?" (ADR-0002)

게임 로직을 UI(유아이) 코드와 분리하기 위해서입니다.
화면을 직접 켜지 않아도 데미지 계산, 상태이상 적용 같은 게임 규칙을
단독으로 테스트할 수 있어야 했습니다.
MVC(엠브이씨)나 MVVM(엠브이브이엠) 같은 방식도 검토했지만,
1인 개발 규모에서 테스트 가능성과 단방향 의존성을 동시에 충족하는 건
4계층 구조가 가장 적합했습니다.

----------------------------------------------------------------


[ Q7 ] "왜 Riverpod(리버팟)을 선택했나요?" (ADR-0003)

테스트마다 상태를 독립적으로 격리할 수 있어서입니다.
전투 화면, 지도 화면, 보상 화면이 같은 데이터를 공유해야 하는데,
Provider(프로바이더)나 GetX(겟엑스) 같은 전역 싱글톤 방식은
테스트 간 데이터가 섞이는 문제가 있었습니다.
Riverpod(리버팟)은 ProviderContainer(프로바이더 컨테이너) 하나만 만들면
Flutter(플러터) 위젯 없이도 상태를 테스트할 수 있어서 선택했습니다.

----------------------------------------------------------------


[ Q8 ] "왜 SharedPreferences(쉐어드 프리퍼런스)를 선택했나요?" (ADR-0004)

저장 데이터가 레벨, XP(엑스피), 해금 카드 목록 정도로 단순하고,
서버 없이 오프라인으로 완전히 동작해야 했기 때문입니다.
Firebase(파이어베이스) 같은 클라우드 DB(디비)도 검토했지만,
과제 프로젝트 특성상 서버 비용을 쓸 수 없고 인터넷 의존도 원하지 않았습니다.
SQLite(에스큐라이트)는 스키마 설계가 필요해 데이터 규모 대비 과했고,
SharedPreferences(쉐어드 프리퍼런스)가 요구에 딱 맞았습니다.

----------------------------------------------------------------


[ Q9 ] "왜 맵을 CustomPaint(커스텀 페인트)로 그렸나요?" (ADR-0005)

맵이 노드와 노드를 잇는 대각선 경로를 표시해야 했기 때문입니다.
Flutter(플러터)의 기본 Row(로우), Column(컬럼) 위젯으로는 대각선을 그릴 수 없고,
서드파티 그래프 라이브러리를 쓰기엔 저희 맵이 9개 노드로 너무 단순했습니다.
CustomPaint(커스텀 페인트)로 직접 그리면 좌표 계산부터 색상·굵기까지
완전히 제어할 수 있어 채택했습니다.
