## Q&A 대비

### Q1. ADR(아키텍처 의사결정 기록) — 무엇을, 왜 선택했나요?

총 5개의 ADR을 작성했고, 핵심은 4가지입니다.

**ADR-0001 — 플랫폼: Flutter**
Android·iOS 동시 지원이 필수인데 1인 개발이라 두 플랫폼을 따로 만들 여력이 없었습니다. React Native는 JS Bridge 때문에 카드 애니메이션 같은 커스텀 UI에 제약이 있고, Android·iOS 단독이나 Kotlin Multiplatform은 양 플랫폼 동시 지원 요구를 충족하지 못합니다. Flutter는 단일 코드베이스로 두 플랫폼을 커버하면서 자체 렌더링 엔진으로 카드 UI를 자유롭게 구현할 수 있고, Hot Reload로 개발 이터레이션도 빨라서 선택했습니다.

**ADR-0002 — 아키텍처: 4-Layer + Riverpod**
게임 로직(데미지 계산, 상태이상, 턴 판정)이 UI와 섞이면 화면 없이는 테스트할 수 없고 TDD도 불가능해집니다. MVC·MVP·MVVM은 Flutter 구조나 Riverpod과 자연스럽게 맞지 않고, Clean Architecture는 카드 한 장 효과에도 UseCase 클래스가 필요해 1인 단기 개발엔 과합니다. 그래서 UseCase를 Application 계층에 통합한 4-Layer(Presentation/Application/Domain/Data) 구조를 채택했습니다.

**ADR-0003 — 상태관리: Riverpod**
전투·맵·보상 화면이 같은 데이터를 공유해야 하는데, Provider는 BuildContext가 필요해 계층 분리가 깨지고, Bloc은 카드 한 장에도 Event→State 변환 코드가 장황해지고, GetX는 전역 싱글톤이라 테스트 격리가 근본적으로 안 됩니다. Riverpod은 BuildContext 없이 ProviderContainer만으로 테스트를 완전히 격리할 수 있어서 선택했습니다.

**ADR-0004 — 영속성: SharedPreferences**
저장할 데이터가 레벨·XP·해금 카드 목록 정도로 단순하고, 서버 비용을 쓸 수 없는 과제 프로젝트라 완전 오프라인 동작이 필요했습니다. Firebase는 인터넷·계정이 필요하고, SQLite는 이 정도 규모엔 스키마 설계가 과해서, 단순한 키-값 저장에 정확히 맞는 SharedPreferences + JSON을 선택했습니다.

**(참고) ADR-0005 — 맵 렌더링: CustomPaint**
던전 맵은 노드 사이를 잇는 대각선 경로와 상태별 색상 표시가 필요한데, Row·Column·Stack 조합으로는 대각선을 그릴 수 없고 탭 위치와 그려지는 위치가 어긋날 위험이 있습니다. 서드파티 그래프 라이브러리는 9개 노드짜리 맵엔 과한 의존성입니다. CustomPaint + CustomPainter로 직접 그리고, `computeNodePositions` 함수를 그리기와 탭 인식 양쪽에 공통으로 써서 항상 일치시켰습니다.

### Q2. 앱의 구조는 어떻게 되어 있나요? (레이어 & 디렉토리 구조)

4개 계층으로 나뉩니다.
- `presentation/` — 화면(위젯)만 담당, 버튼 이벤트를 Application에 전달
- `application/` — Riverpod Notifier가 상태 소유 + 명령 처리 (Use Case 겸임)
- `domain/` — 순수 Dart 게임 규칙 (Flutter·Riverpod import 금지)
- `data/` — SharedPreferences 읽기/쓰기 캡슐화

의존 방향은 `Presentation → Application → Domain ← Data`로 단방향이고, 역방향 import는 금지입니다. 덕분에 화면 디자인을 바꿔도 게임 규칙엔 영향이 없고, `domain/`은 위젯이나 ProviderContainer 없이도 순수 Dart 테스트로 단독 검증할 수 있습니다.

> 비유로 설명하면 세 가지 역할입니다. BattleEngine(Domain)은 주방 — HP가 얼마나 깎이는지 실제 숫자를 계산합니다. BattleNotifier(Application)는 서빙 직원 — 계산 결과를 화면으로 가져다줍니다. BattleScreen(Presentation)은 손님 — 받은 숫자를 그냥 화면에 보여줍니다. `ref.watch`는 손님이 "숫자 바뀌면 알려주세요"라고 미리 등록해두는 것입니다.

### Q3. 개발 환경 설정은 어떻게 했나요?

1. Flutter SDK 설치 (`flutter doctor`로 확인)
2. Android 빌드용 JDK 17 설치
3. 저장소 클론 후 `flutter pub get`으로 의존성 설치
4. `flutter doctor -v`, `flutter devices`로 환경 확인
5. `flutter run -d <device-id>`로 첫 실행

5분 안에 동일한 환경을 구성할 수 있습니다. 한 가지 주의점은, Web(Chrome)에서 실행할 때 매번 임시 프로필로 켜지면 `localStorage`가 초기화돼 세이브 데이터가 날아가기 때문에, 고정 프로필로 Chrome을 실행하도록 별도 스크립트를 구성해 두었습니다.

### Q4. 빌드와 배포는 어떤 단계로 이루어지나요?

빌드는 "실행 가능한 산출물을 만드는 것", 배포는 "그 산출물을 사용자에게 전달하는 것"입니다.

- **Android**: `flutter build apk` 또는 `flutter build appbundle` → APK/AAB 생성 (현재는 debug 서명 사용, 정식 서명·스토어 등록은 이번 범위 밖)
- **iOS**: `flutter build ipa` → IPA 생성 (Xcode에서 Signing & Capabilities 설정 필요)
- **Web**: `flutter build web` → 정적 파일 생성, 이 결과물(`docs/`)은 GitHub Actions가 GitHub Pages에 자동 배포

즉 세 플랫폼 모두 빌드는 실제로 가능하고, 배포까지 자동화된 건 현재 Web(프로젝트 웹페이지) 한 가지입니다.

### Q5. 어디까지 AI가 만들었고, 직접 작성한 부분은 무엇인가요?

AI는 코드 초안 작성, 문서화, 아키텍처 설계 검토에 적극 활용했습니다. 다만 AI가 만든 코드라도 항상 테스트를 먼저 작성하고, 그 테스트를 통과해야만 병합하는 TDD 방식으로 검증했습니다.

직접 한 부분은 게임 규칙과 아키텍처 구조 설계, ADR에 담긴 선택과 그 근거, 카드 데미지·상태이상 계산 로직의 설계와 검토입니다. AI가 코드 초안을 작성해도 무엇을 만들지, 어떻게 연결할지, 왜 그 방식을 택할지는 저희가 직접 판단하고 결정했습니다.

### 5p — "저희만의 기법"이 정확히 무엇인가요?

카드 30장 각각에 "이건 공격 카드", "이건 방어 카드", "이건 회복 카드"라는 분류 스티커를 붙여놨다고 생각하면 됩니다 (`CardEffectType`: damage/block/heal/buff/draw 등).

이 스티커가 있으면, "공격 스티커가 붙은 카드를 쓰면 내 힘(strength)만큼 데미지가 자동으로 추가된다"처럼 카드 전체에 적용되는 공통 규칙을 딱 한 번만 적어두면 됩니다. 30장마다 따로 적을 필요가 없습니다.

그래서 새 카드를 만들 때는
- "공격" 스티커만 붙이면 → 힘 보너스, 집중(Focus) 보너스 같은 공통 효과는 자동 적용
- 그 카드만의 특별한 동작(예: "적에게 취약 부여")만 한 줄(case 분기) 추가하면 끝

"분기 하나만 추가하면 확장된다"는 게 바로 이 의미입니다. 실제 코드는 `lib/domain/battle_engine.dart`의 `switch (card.type)` 안에 새 `case` 하나만 추가하면, 힘/집중 보너스는 `effectType == CardEffectType.damage` 체크에서 이미 처리됩니다.

### 9p — 빌드/배포, 실제로 되는 건가요?

- 빌드는 실제로 됩니다. `flutter build apk` / `ipa` / `web` 명령으로 Android·iOS·Web 산출물을 만들 수 있고, Flutter 표준 기능으로 동작합니다.
- "자동 배포"는 프로젝트 웹페이지(이 발표자료·WBS 페이지)에 한정됩니다. `.github/workflows/pages.yml` 워크플로우 하나가 `docs/` 폴더를 GitHub Pages에 자동으로 올려줄 뿐, 앱(APK/IPA)을 스토어에 배포하는 파이프라인은 없습니다.
- 즉 "Android·iOS·Web으로 빌드는 가능하고, 프로젝트 웹사이트는 GitHub Actions로 자동 배포된다"가 정확한 표현입니다.

### 10p — "절차적 맵 생성은 100회 이상 반복 검증해야 신뢰할 수 있다"는 무슨 뜻인가요?

- 절차적 생성 = 사람이 맵을 미리 그려두는 게 아니라, 코드가 매번 랜덤으로 맵을 새로 만든다는 뜻입니다.
- 문제는 "랜덤"이라서, 어쩌다 한 번은 운 나쁘게 이상한 맵이 나올 수도 있습니다 (예: 어떤 칸에서 보스로 가는 길이 막혀버림).
- 딱 한 번 테스트해서 괜찮았다고 "이 코드는 항상 안전하다"고 말할 수 없습니다 — 그날 랜덤 운이 좋았을 수도 있기 때문입니다.
- 그래서 테스트 코드에 랜덤 시작값(seed) 10개를 미리 정해두고 (`_seeds = [0, 7, 42, 100, 777, ...]`), 그 10개 각각으로 맵을 만들어 "층이 12개 다 있나?", "막힌 길은 없나?" 같은 규칙들을 검사합니다. 검사 항목이 여러 개라서 총 검증 횟수가 100번이 넘습니다.

한 줄 요약: 랜덤으로 만드는 거라서, 여러 번(다른 시드로) 만들어도 항상 규칙을 지키는지 확인해야 진짜 믿을 수 있다는 뜻입니다.

### 11p — 성능 최적화 & 코드 품질 관리

두 가지를 한 문장에 합쳐놓은 것이라 따로 풀어서 설명합니다.

**① 성능 (던전 맵 → CustomPaint → 60fps)**
- 던전 맵에는 9개의 칸(노드)과 그 사이를 잇는 선들이 있습니다.
- Flutter에서 화면은 보통 버튼·이미지 같은 "위젯"을 여러 개 쌓아서 만드는데, 위젯이 많아지면 느려지고, 위젯으로는 비스듬한 선(대각선)을 긋기도 어렵습니다.
- CustomPaint = 도화지에 직접 그림을 그리는 기능. 위젯을 쌓는 대신 코드로 직접 점과 선을 그려서 위젯 개수를 줄입니다.
- 60fps = 화면이 1초에 60번 새로 그려진다는 뜻. 이 정도면 사람 눈에 끊기지 않고 매끄럽게 보입니다.

**② 코드 품질 (AGENTS.md + pre-commit 훅)**
- AGENTS.md = 코드 짤 때 지켜야 할 약속 모음 문서입니다 (예: 모든 함수에 설명 주석 달기, 숫자를 그대로 쓰지 말고 이름 붙이기 등).
- pre-commit 훅 = 저장(커밋)하기 직전에 자동으로 실행되는 검사기. `flutter analyze`(코드 실수/경고 검사)와 `flutter test`(테스트 446개 통과 검사)를 자동으로 실행합니다.
- "경고 0건을 강제"한다는 것은, 이 검사를 통과하지 못하면 커밋 자체가 안 된다는 뜻입니다. 그래서 항상 깨끗한 코드 상태를 유지하도록 강제하는 장치입니다.
