## Slide 2 — 문제 정의 & 비전 (약 28초)

일단, 덱빌딩 로그라이크의 핵심 재미는 *"이번 런엔 어떤 빌드로?"*라고 설명할 수 있겠는데요.
그런데 모바일에서는 이 재미를 가볍게 즐길 수가 없습니다.

왜냐하면 모바일 게임 대부분이 Pay-to-Win 과금 구조라 순수 게임플레이에 집중하기 어렵고,
카드와 시스템이 복잡해 진입 장벽이 높으며, 메타 성장이 없으면 반복 플레이 동기도 약합니다.

그래서 저희는 과금 없이 플레이만으로 모든 콘텐츠를 해금하는 구조, 미니멀 규칙으로 진입 장벽을 낮추고,
런마다 XP를 쌓아 카드와 유물이 영구 해금되는 메타 성장으로 이 세 가지를 해결합니다.

## Slide 5 — 아키텍처 설계 (약 32초)

아키텍처는 4계층으로 분리했습니다.

Presentation은 화면만, Application의 Riverpod Notifier(리버팟 노티파이어)가 상태를 관리하고,
Domain은 순수 Dart로 게임 로직, Data는 SharedPreferences(쉐어드 프리퍼런스)를 통해 데이터를 저장하고 불러옵니다.

의존 방향은 Presentation이 Application을, Application이 Domain과 Data를 각각 의존합니다.
Domain에는 Flutter와 Riverpod import를 금지해 테스트 가능성을 극대화했습니다.

덕분에 화면 없이 ProviderContainer(프로바이더 컨테이너) 하나만으로 전체 게임 로직을 단독 테스트할 수 있습니다.

## Q&A 단골 질문 대비

---

### Q1. "왜 Flutter를 선택했나요?"

Android와 iOS를 단일 코드베이스로 동시에 지원할 수 있어서입니다.
게임 로직은 플랫폼과 무관하게 동일하고, UI만 Flutter 위젯으로 구성하면 되기 때문에
두 플랫폼을 따로 개발하는 비용 없이 모바일 타깃을 달성할 수 있었습니다.

---

### Q2. "이 화면의 데이터는 어디서 오나요?"

Presentation 계층의 위젯이 Riverpod의 ref.watch로 Provider를 구독합니다.
예를 들어 전투 화면은 battleProvider를 구독해서, BattleNotifier가 상태를 바꾸면
위젯이 자동으로 다시 그려지는 구조입니다.
데이터 자체는 Domain의 BattleEngine이 계산하고, Application이 이를 받아 상태로 emit합니다.

---

### Q3. "지금 가장 큰 문제는?"

현재 UI와 ViewModel의 전체 통합 작업이 진행 중입니다.
각 계층은 완성됐지만 end-to-end로 연결해 실제 게임 루프가 동작하는지 검증하는 단계입니다.
이후 테스트 커버리지 80% 확보와 QA가 남아 있습니다.

---

### Q4. "어디서부터 어디까지 AI가 만들었나요?"

AI를 적극적으로 활용했습니다. 문서 작성, 코드 초안 생성, 아키텍처 설계 검토에 사용했습니다.
다만 AI가 생성한 코드는 반드시 테스트를 먼저 작성하고, 테스트를 통과할 때만 병합하는
TDD 방식으로 검증했습니다. 이해하지 못한 코드는 병합하지 않았습니다.

---

### Q5. "본인이 직접 짠 부분은 어떤 거예요?"

게임 규칙과 아키텍처 구조를 직접 설계했고,
카드를 쓸 때 데미지가 어떻게 계산되는지, 상태이상이 어떻게 적용되는지 로직을 직접 짰습니다.
AI가 코드 초안을 써줬지만, 무엇을 만들지, 어떻게 연결할지는 저희가 판단하고 결정했습니다.

---

### Q6. "왜 4계층 아키텍처를 선택했나요?" (ADR-0002)

게임 로직을 UI 코드와 분리하기 위해서입니다.
화면을 직접 켜지 않아도 데미지 계산, 상태이상 적용 같은 게임 규칙을 단독으로 테스트할 수 있어야 했습니다.
MVC나 MVVM 같은 방식도 검토했지만, 1인 개발 규모에서 테스트 가능성과 단방향 의존성을 동시에 충족하는 건 4계층 구조가 가장 적합했습니다.

---

### Q7. "왜 Riverpod을 선택했나요?" (ADR-0003)

테스트마다 상태를 독립적으로 격리할 수 있어서입니다.
전투 화면, 지도 화면, 보상 화면이 같은 데이터를 공유해야 하는데,
Provider나 GetX 같은 전역 싱글톤 방식은 테스트 간 데이터가 섞이는 문제가 있었습니다.
Riverpod은 ProviderContainer(프로바이더 컨테이너) 하나만 만들면 Flutter 위젯 없이도 상태를 테스트할 수 있어서 선택했습니다.

---

### Q8. "왜 SharedPreferences를 선택했나요?" (ADR-0004)

저장 데이터가 레벨, XP, 해금 카드 목록 정도로 단순하고, 서버 없이 오프라인으로 완전히 동작해야 했기 때문입니다.
Firebase 같은 클라우드 DB도 검토했지만, 과제 프로젝트 특성상 서버 비용을 쓸 수 없고 인터넷 의존도 원하지 않았습니다.
SQLite는 스키마 설계가 필요해 데이터 규모 대비 과했고, SharedPreferences가 요구에 딱 맞았습니다.

---

### Q9. "왜 맵을 CustomPaint로 그렸나요?" (ADR-0005)

맵이 노드와 노드를 잇는 대각선 경로를 표시해야 했기 때문입니다.
Flutter의 기본 Row, Column 위젯으로는 대각선을 그릴 수 없고,
서드파티 그래프 라이브러리를 쓰기엔 저희 맵이 9개 노드로 너무 단순했습니다.
CustomPaint로 직접 그리면 좌표 계산부터 색상·굵기까지 완전히 제어할 수 있어 채택했습니다.
