## AI가 한 것

카드·유물·레벨 시스템을 추가하는 기능 PR(`f9ed116`, `c56bca9` 등)을 진행하면서 **코드는 매번 갱신했지만, SPECS.md·README.md 등 문서는 갱신하지 않았다.**
그 결과:

- SPECS.md의 레벨업 해금 표는 "예시"로 작성된 자리표시자(레벨 1~5, "5+ 계속 확장")였고, 실제 `MetaProgress.levelUnlocks` / `xpThresholds` 상수(레벨 1~10)와 전혀 달랐다.
- "레벨업 시 카드와 유물이 함께 해금된다"는 서술이 있었지만, 실제 코드에서 유물 20종은 레벨과 무관하게 처음부터 보상 풀에 포함된다.
- 신규 카드 4종(극한 각성·업화·철옹성·폭격 태세)이 SPECS.md 카드 표에 아예 없었고, 유물 20종을 정리한 섹션 자체가 없었다.
- README의 테스트 뱃지(424)와 CLAUDE.md의 테스트 디렉토리 목록(6개)도 실제(431개, 16개)보다 뒤처져 있었다.

## 무엇이 틀렸는가

AI는 "기능 구현"을 요청받으면 코드와 테스트는 갱신하지만, **명시적으로 요청하지 않는 한 관련 기획/명세 문서까지 동기화하지 않는다.** 코드와 문서가 서로 다른 파일에 분리되어 있으면, 매 기능 추가마다 작은 불일치가 누적되어 결국 "문서가 코드를 설명하지 못하는" 상태(documentation drift)가 된다.

## 내가 수정한 것

"코드베이스 전수 검토 후 MD 파일 정확성을 점검해줘"라는 별도 프롬프트를 명시적으로 요청했다. 그 결과:

- SPECS.md에 11장 "유물(Relics)" 섹션을 신설해 20종 효과를 표로 정리
- 레벨 1~10 해금표를 `MetaProgress` 상수 기준으로 전면 재작성, 잘못된 "유물도 레벨업으로 해금" 서술 삭제
- 누락된 신규 카드 4종을 카드 표에 추가
- README 테스트 뱃지, CLAUDE.md 테스트 디렉토리 목록, docs/setup.md·testing.md의 테스트 현황을 431/431로 갱신

## 배운 것

> "기능 추가해줘"와 "문서도 함께 갱신해줘"는 AI에게 **별개의 요청**이다.

기능 단위 PR마다 문서 동기화를 매번 요구하는 대신, 일정 주기로 "코드베이스 전수 검토 후 문서 정확성 검증"이라는 별도 점검 프롬프트를 던지는 것이 효율적이다. 이 프로젝트에서는 `BONUS.md`/발표 준비 시점이 자연스러운 점검 트리거가 되었다.
