## AI가 한 것

`064ca61`에서 블록을 뚫고 HP가 감소하면 화면 가장자리에 빨간 `RadialGradient` 오버레이를 500ms 표시하는 기능을 추가했다. 이후 세 번에 걸쳐 같은 위젯의 `radius`/`stops`/`opacity` 값을 다음과 같이 왕복 조정했다.

| 커밋 | radius | stops | opacity |
|------|--------|-------|---------|
| 56c8f24 | 1.4 | `[0.0, 0.50, ...]` | 가시성 확보 위해 상향 |
| 43a7192 | (유지) | 0.76 → **0.60** | 0x66/0xAA → **0xAA/0xDD** (상향) |
| 26e7b77 | 1.0 → **1.5** | 0.55 → **0.76** (다시 좁게) | 0x88/0xCC → **0x66/0xAA** (다시 하향) |

`43a7192`에서 올렸던 opacity를 `26e7b77`에서 다시 낮추는 등, 같은 파라미터가 두 커밋 사이에서 왕복했다.

## 무엇이 틀렸는가

"가시성이 떨어진다", "너무 크다", "가장자리에만 좁게" 같은 **상대적·정성적 피드백만으로는 AI가 목표 수치를 가늠할 수 없다.** AI는 매번 그럴듯한 값을 추측해서 바꾸고, 다음 피드백에서 그 추측이 틀렸음이 드러나면 다시 반대 방향으로 추측한다 — 즉 이진 탐색이 아니라 **무작위 왕복**에 가깝다.

## 배운 것

> 비주얼/UI 피드백은 "느낌"이 아니라 **목표 수치 또는 스크린샷**으로 전달해야 왕복 수정을 줄일 수 있다.

예를 들어 "가장자리 20%만 빨갛게, 중앙 80%는 완전 투명, 최대 opacity 0xAA" 처럼 구체적 비율/값을 먼저 제시하면 AI가 추측 대신 계산으로 값을 도출한다. 다음에 비주얼 버그를 수정할 때는 (1) 스크린샷 첨부 또는 (2) 목표 비율을 숫자로 명시 — 둘 중 하나를 우선 시도한다.
