## AI가 한 것

`dart:io` 기반 저장 로직을 `shared_preferences`로 전환해 Flutter Web(Chrome) 빌드를 통과시키고, 세이브 슬롯 3개 저장·로드·삭제 기능을 구현했다. 빌드는 성공했고 `flutter test`도 모두 통과했다.

## 무엇이 틀렸는가

`flutter run -d chrome` 기본 명령어로 실행했을 때, **매 실행마다 새 임시 Chrome 프로필이 생성**되어 `localStorage`(세이브 슬롯·레벨·XP)가 초기화되었다. 처음에는 "세이브 기능이 작동하지 않는다"는 버그로 보였지만, 실제로는 코드가 아니라 **Web 디버그 모드 특유의 실행 환경 문제**였다.

AI는 "Web 빌드가 컴파일되고 실행된다"까지만 확인했고, **개발자가 반복 실행하며 데이터가 유지되는 일상적 워크플로우**는 별도로 검증하지 않았다. 이 문제는 Android/iOS 실기기에서는 애초에 발생하지 않기 때문에, "Web 지원 추가"라는 작업 범위 안에서는 자연스럽게 누락되기 쉬운 항목이었다.

## 내가 수정한 것

- `--user-data-dir`로 고정 Chrome 프로필 경로를 지정하는 `run_web.ps1` 작성
- `.vscode/launch.json`에 "Flutter Web (Chrome, 데이터 유지)" 구성 추가
- `docs/setup.md`에 "왜 이렇게 해야 하는가"를 포함한 경고 박스 추가 (배포 빌드에서는 문제없음을 명시)

## 배운 것

> 새 플랫폼(Web/Desktop) 지원을 추가할 때 "빌드 성공 + 테스트 통과"는 **출발점일 뿐**이다. 그 플랫폼에서 **개발자가 반복적으로 작업할 때 실제로 부딪히는 환경 차이**(프로필 격리, 파일시스템 권한, 포트 충돌 등)는 직접 그 플랫폼에서 몇 번 실행해봐야 드러난다.

이후 새 플랫폼 작업에는 "기능 구현 + 테스트 통과" 외에 "그 플랫폼에서 2회 이상 반복 실행해 상태 유지를 확인"을 체크리스트에 추가했다.
