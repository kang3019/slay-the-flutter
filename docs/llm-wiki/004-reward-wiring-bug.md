| 항목 | 내용 |
|------|------|

| **카테고리** | 실패 사례와 원인 |
| **발견 경위** | 신규 카드 18장을 보상 풀에 추가한 뒤 런을 처음부터 끝까지 직접 플레이하다가, 보상으로 받은 카드·회복한 HP·갱신된 덱이 다음 전투에 전혀 반영되지 않는 것을 발견 |

## AI가 한 것

`_rewardPool`에 신규 카드 18장을 추가해 보상 화면에 새 카드가 등장하도록 구현했다. 단위 테스트(`reward_test` 류)도 보상 풀에서 카드가 정상적으로 뽑히는지는 검증해 통과했다.

하지만 `BattleEngine.start()`는 여전히 **고정된 기본 덱과 초기 HP로만** 전투를 생성했다 — 보상으로 얻은 카드가 담긴 `RunState.deck`이나 회복된 `playerHp`를 받는 파라미터가 없었다.

## 무엇이 틀렸는가

이 프로젝트는 "보상 데이터를 만드는 곳"(보상 화면/풀)과 "그 데이터를 실제로 사용하는 곳"(`BattleEngine.start()` 진입점)이 분리되어 있다. 새 보상 카드를 추가하는 작업은 전자만 건드리면 컴파일도 되고 단위 테스트도 통과하지만, **후자의 파라미터를 함께 갱신하지 않으면 "추가됐지만 적용되지 않는" 상태**가 된다. AI는 "보상 풀에 카드를 추가해줘"라는 요청의 범위를 풀 자체로 한정했고, 그 데이터가 다음 전투 생성 호출까지 도달하는 전체 경로는 별도로 점검하지 않았다.

## 내가 수정한 것

런 전체를 처음부터 끝까지 직접 플레이하며 발견한 뒤:

- `BattleEngine.start()`에 `cards`, `playerHp` 파라미터 추가
- `app_router`에서 `RunState.deck`, `RunState.playerHp`를 `BattleEngine.start()`로 전달하도록 수정
- 전투 화면 상단에 보유 유물 칩 목록을 추가해, 런 상태가 전투에 실제로 반영되는지 시각적으로 바로 확인 가능하게 함

## 배운 것

> 단위 테스트는 "함수 단위 정합성"을 검증하지만, **새 데이터가 실제 진입점까지 도달하는 연결**은 검증하지 못한다.

새로운 데이터(카드/유물/이벤트 등)를 추가하는 PR은 머지 전에 "런을 처음부터 끝까지 직접 플레이"하는 수동 점검을 거쳐야 한다. 이 프로젝트의 `/verify` 워크플로우(앱을 실제로 실행해 동작을 확인)를 보상/진행 관련 변경에는 항상 적용하기로 했다.
