## 위험 등록부

| # | 카테고리 | 위험 내용 | 발생 가능성 | 영향도 | 대응 방안 |
|---|----------|-----------|-------------|--------|-----------|
| R-01 | **기술** | Riverpod·StateNotifier 등 처음 다루는 상태 관리 라이브러리의 학습 곡선으로 인해 ViewModel 구현이 예상보다 2~3배 오래 걸릴 수 있음 | 높음 | 높음 | 공식 문서 + 샘플 프로젝트 1개를 Week 1에 미리 완독. 모르는 패턴은 구현 전에 ADR로 결정 이유를 문서화하여 재학습 비용 절감 |
| R-02 | **일정** | 중간고사·기말고사·타 과목 과제 폭주 주간에 개발이 멈춰 마일스톤이 밀릴 수 있음 | 높음 | 중간 | WBS(02-wbs.json)에 버퍼 주간을 사전 배치. 폭주 주간 전에 Must Have 기능을 선완료하는 것을 목표로 하며, 지연 발생 시 Could Have를 즉시 스코프 밖으로 이동 |
| R-03 | **협업** | 1인 팀 특성상 의사결정 지연 없이 혼자 모든 설계·구현·테스트를 수행해야 하므로 멘탈 관리와 작업량 부담이 누적될 수 있음 | 중간 | 중간 | 매주 금요일 15분 자기 리뷰(WBS 체크 + 일주일 회고) 루틴 유지. 하루 개발 시간 상한을 미리 설정해 번아웃 방지. 막히는 부분은 24시간 이내 교수/TA에게 질문 |
| R-04 | **외부 의존** | SharedPreferences 등 로컬 저장소 API 변경이나 무료 티어 제한(Firebase 등 클라우드 사용 시)으로 데이터 레이어가 깨질 수 있음 | 낮음 | 높음 | **로컬 우선(Local-First) 아키텍처** 채택: 외부 서비스 의존 없이 SharedPreferences + JSON 직렬화로 모든 영속성 구현. 클라우드는 Won't Have로 고정 |
| R-05 | **AI 의존** | AI가 생성한 복잡한 턴제 전투 로직을 개발자 본인이 완전히 이해하지 못한 채 코드베이스에 병합하면, 버그 발생 시 디버깅이 불가능해지고 평가에서 "본인 코드가 아님"으로 판정될 수 있음 | 중간 | 매우 높음 | **TDD 강제**: AI 코드 생성 전 테스트 케이스를 먼저 작성하고, 생성된 코드가 테스트를 통과할 때만 병합. 핵심 설계 결정마다 ADR 작성. 이해하지 못한 코드 블록은 `// TODO(kang3019): 이해 필요` 로 표시 후 리팩터링 |

## 위험 수준 정의

| 발생 가능성 | 기준 |
|-------------|------|
| 높음 | 이번 학기 기준 발생할 가능성이 50% 이상 |
| 중간 | 20~50% |
| 낮음 | 20% 미만 |

| 영향도 | 기준 |
|--------|------|
| 매우 높음 | 프로젝트 평가에 직접적 감점 또는 제출 불가 |
| 높음 | 핵심 기능 1개 이상 미완성 |
| 중간 | 일정 1주 이상 지연 또는 Should Have 기능 탈락 |
| 낮음 | 일정 3일 이내 영향, 회복 가능 |

## 위험 모니터링 주기

매주 금요일 WBS 리뷰 시 위험 등록부를 함께 검토하고, 상태 변화(발생·해소·추가)가 있으면 이 파일을 업데이트한다.
